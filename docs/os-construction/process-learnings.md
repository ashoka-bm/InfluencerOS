# Process Learnings

This file stores repo-level learnings about building InfluencerOS and improving its skills.

Do not store creator performance learnings here. Creator learnings belong in:

```text
workspace-library/creators/<creator-slug>/memory/
```

## Promotion Rule

1. Capture scope-specific feedback in the relevant local context:
   - Creator feedback in the Creator Workspace.
   - InfluencerOS first-party feedback in `context/learnings.md` or this file.
   - Skill-specific local feedback in `SKILL.local.md` when it changes invocation rules.
2. Watch for repeated feedback across projects or creators.
3. Promote only repeatable, general rules into the base skill.
4. Record promoted rules in the skill registry or an ADR when they affect architecture.

## Current Learnings

- InfluencerOS needs both first-party OS persona memory and creator-specific memory.
- Skill feedback must stay scoped until repeated evidence shows it belongs in the core skill.
- 2026-07-03: Fixture-driven "no warnings" tests bless whatever the fixture does not exercise — the slice 3 promotion-gate fixture cited a video pack no test created, so the gap passed as zero warnings. When a validator gains a new ref type, add an unresolvable-fixture probe in the same change.
- 2026-07-03: Deprecation sweeps must grep prose vocabulary, not just schema tokens. The README still taught the five-ideas flow after slice 3's token-based sweep passed; check the flow narrative in README/ARCHITECTURE against CONTEXT.md whenever a pipeline concept is replaced.
- 2026-07-03: Every copy of a closed vocabulary needs a drift pin — schema embeddings, code constants, and test canonicals alike. The slice 3 reviews found the only unpinned enum copies (project schema, then code-level platform and format sets); the drift test now asserts all three layers agree.
- 2026-07-03: Test scaffolds must mirror constructor output, not a convenient approximation. The research scaffold placed projects in id-named folders while `init-project` creates slug-named ones, so the warning-target check's id-folder assumption passed for a layout no constructor produces; slice 5 caught it only because a new check scanned by manifest id. When a scaffold hand-builds what a CLI constructs, build it through the CLI or copy the CLI's layout exactly.
- 2026-07-03: Batch plans should put a skill folder and its registry/context-matrix rows in the same batch — the registry drift checks pin them to one commit, so a "skill first, docs later" split cannot land green. The slice 5 plan split them across batches B and C and had to fold the rows into B.
- 2026-07-03: When the same record type is reachable from more than one CLI command, its checks must live in shared functions called by every path. The queue/research promotion checks accumulated one slice at a time on whichever command was closest, and the paths drifted until the five-slice review reproduced four one-sided bypasses; per-slice reviews audit the slice, not the seams between slices.
- 2026-07-03: Closing a vulnerability class at the validation seam does not close it at the write seam. The slice 1 review added symlink containment to `validate workspace`; the same broken-symlink class stayed open in `import-intake`'s copy until the five-slice review — when a review closes a containment gap, sweep every filesystem write for the same class in the same change.
- 2026-07-03: Verification scripts recorded in progress docs are prose, not pinned — they rot the moment a new at-rest check lands (slice 5's closure check broke the documented queue-before-project ordering silently). Re-run the documented script end to end whenever a slice adds a cross-record check, not only the tests.
- 2026-07-04: Mutator-only invariants are not enough for file-first records. Slice 7's registration command enforced output-package format/project matching, but `validate project` did not, so a hand-edited packaged project validated at rest. Any invariant a writer enforces must also be re-checkable by the standing validator for that record state.
- 2026-07-04: Schema/ADR additions need an inventory-doc sweep after tests pass. The research-intelligence hardening added two schemas and ADR 0021 cleanly, but architecture maps, README, and pipeline-contract counts still described the old record set until a stale-reference scan caught them.
- 2026-07-04: A reconciliation check must run in both directions, and a new ledger must inherit its sibling ledger's guards. The `yield_stats` reconciliation checked records->sources but not sources->records, so a hand-edited source could advertise an invented usefulness history with zero backing records — the exact "writer enforces, at-rest validator does not re-check" class flagged one slice earlier for output packages. The same review found `source-yield.jsonl` was missing the duplicate-id guard the evidence/metric ledgers carry, and enforced the gated-access-method prohibition on the search plan but not on the yield record that attests what the run actually did. When a slice adds a ledger file or an aggregate derived from records, port every sibling ledger's guard, reconcile the aggregate over the full record set (zeros included), and enforce prohibitions on the record of what happened, not only the record of what was planned.
- 2026-07-04: Gate advisory checks on durable at-rest outputs, not a mutable declarative flag. The thin-evidence WARN first keyed only on `research-run.json.material_update`, so a run with real `outputs.finding_ids`/`idea_queue_entry_ids` but a stale `material_update: false` slipped the check; deriving "material" from the declared outputs closed it. A boolean an author sets by hand is weaker evidence than the id lists the record already carries.
- 2026-07-04: A guidance rule needs its canonical example to model it, and needs a structured field to be auditable. The anti-contamination query rule shipped as prose while `examples/research-search-plan.example.json` still used "Agentic OS trend/discussion pattern" as `routing_basis` — the reference example taught the very contamination the rule forbade. Free-text `routing_basis` (checked only for non-emptiness) also made the rule un-enforceable, so a required `term_basis` enum was added. When adding a rule meant to be auditable: fix the exemplar in the same change, and prefer a closed-set field the validator checks over prose plus an author's promise.
- 2026-07-05: A standing-approval carve-out written against an access-method class silently re-approves every adapter in that class. ADR 0022's first cut exempted all `api_backed`/`scraping_api` adapters, which would have let `youtube_data_api` validate as active/use_now/approval-false despite the registry saying it still needs approval. Pin an exemption to the exact approved adapter IDs (plus their expected access method), on every enforcement point, not to the capability class.
- 2026-07-05: A secret passed as a URL query param leaks through request logging. The LinkedIn connector put the Apify token in `?token=...` while the HTTP client logged the full URL under a debug flag. Pass provider secrets as Authorization headers and redact query strings before logging — close it at both the caller (header) and the logger (redaction).
- 2026-07-05: A record schema that mirrors an upstream record's field must port the upstream's nullability rules and format gates in the same change — and its tests need one fixture per format class. Slice 1's PublishedPostRecord required a string thumbnail while the Output Package it attests allows null for text formats, so honest article publications failed schema validation; the gap survived because every slice test used the visual example fixture. When a downstream record cites an upstream field, diff the two schemas' type unions before shipping, and probe with both a visual and a text fixture.
- 2026-07-05: A status-conditioned invariant written as equality lapses when the status advances. The packaged-requires-upload-ready check keyed on `status == "packaged"`, so publishing a project silently stopped enforcing it; `_status_at_least` was already available. Write lifecycle invariants as "at or past X", never "exactly X", unless the invariant genuinely ends.
- 2026-07-05: A consistency rule implemented inside an optional derivation branch only fires when derivation runs. Slice 2's pre-publication rejection lived in `_derive_hours_since_publish`, so supplying `hours_since_publish` skipped the check entirely. Cross-field consistency checks belong in the shared match seam that every path (and the at-rest validator) calls, never in a compute-if-absent helper.
- 2026-07-05: Python's `json.loads` accepts `NaN`/`Infinity` by default, and every comparison with NaN is false — so schema min/max checks silently pass and `json.dumps` happily re-emits invalid JSON. Reject non-finite numbers in the validator's number branch (fail-closed, all schemas), refuse them at ingestion parsers, and write with `allow_nan=False`.
- 2026-07-05: Containment must be checked against the narrowest root the record declares, not the broadest safe root. The raw-ref check pinned the declared string under `analytics/raw/` but resolved containment against the project root, so a symlink inside `raw/` could point at any project file. When a field promises "under X/", resolve against X, not against "somewhere safe".
- 2026-07-05: Mirroring a proven external implementation carries its shortcuts too. The Agentic OS reference the connectors were modeled on accepted any URL containing `reddit.com` and any URL as an X candidate; copied faithfully, that admits `notreddit.com`, `developers.reddit.com`, and profile URLs as thread/status evidence. When adapting reference code, re-audit its input-trust boundaries against our provenance needs rather than inheriting them.
- 2026-07-05: Resolving two related id lists independently does not validate the relation between them. The summary's cited posts and cited snapshots each resolved to real records, but nothing required a cited snapshot's parent post to be among the cited posts, so multi-publication projects could attribute one post's metrics to another's URL. When a record cites both sides of a parent/child relation, validate the join, not just membership on each side.
- 2026-07-05: A schema-required field that reads as an evidence reference must be validated as one in the same slice that ships it. `source_material_refs` was required prose-shaped provenance, so `../../outside` and missing paths passed at rest — the existing "unresolvable-fixture probe for every new ref type" rule applies to every ref-shaped field on a new record, not only the fields the slice plan names.
- 2026-07-06: Existence of an id is not existence of a record. Slice 4's evidence resolver accepted any JSON under `projects/` that carried the right id field, so a planted `{"performance_summary_id": ...}` file resolved as lesson evidence and validated at rest. An id-resolution seam built for provenance must schema-validate each candidate (record semantics included) and anchor it to its parent manifest before accepting the id — matching an id string is a lookup, not a resolution.
- 2026-07-06: A judgment enum whose contract defines a countable property must count it, or the doc must say the term is prose-only. `multi_post_pattern` was pinned to the schema enum but never evidence-checked, so a single-post lesson could claim a multi-post pattern the skill contract explicitly forbids. When a closed vocabulary term means "at least N of X", enforce N at the writer and the at-rest validator; enum membership alone validates spelling, not meaning.
- 2026-07-06: Section-scoped markdown validation must handle duplicate headings fail-closed. The Creator Lessons checks located the first `## Creator Lessons` heading and stopped at the next `## `, so a second identical section escaped both the writer's dedup and the at-rest validator entirely. When a validator scopes to a named section, either scan every matching section or reject duplicate headings outright.
- 2026-07-06: A later slice's plan text rots against earlier slices' as-built layout. The Phase 2 plan (written before slice 2 landed) specified the slice 5 index scan as `analytics/*.json`, but slice 2 built `analytics/snapshots/` beside `analytics/raw/` — following the plan verbatim would have indexed raw payloads or missed snapshots entirely. Before implementing a slice that names concrete paths or globs, re-derive them from the writer code and its path constants, not from the plan prose; the writer layout is the contract.
- 2026-07-06: A recorded learning class must be swept against every new seam of the same shape, not just the seam that produced it. Slice 5's index scans re-created the slice 4 "existence of an id is not existence of a record" gap one day after it was recorded — the learning was applied to the memory-module resolver but not carried to the next id-consuming scan built the following day. Before closing a slice that adds an id-consuming or ref-resolving seam, check the seam against the recorded learning classes (id-vs-record, ref probes, containment roots, both-direction reconciliation), not only the slice plan's test list.
- 2026-07-06: Cross-record timestamp checks must reject mixed timezone awareness before trusting supplied durations. The Phase 2 full review found a pre-publication analytics snapshot could pass when `snapshot_at` was naive, `published_at` carried `Z`, and `hours_since_publish` was supplied; the shared seam now fails closed on mixed aware/naive timestamp pairs. When a duration field can be supplied or derived, the chronological invariant belongs in the shared match check and must first prove the timestamps are comparable.
- 2026-07-06: An exit criterion that names a CLI path must be probed at that named path, not only at the function seam that implements the rule. The Creative Direction slice 1 carry-forward check lived in the promotion gate and was reachable from `validate research`, but the plan's criterion named `validate workspace`, which never called it — the rule was real and the promised path was dead. When a plan names the command that fails, add a path-level test invoking exactly that command.
- 2026-07-06: Validation gated on a lifecycle status lets premature records dodge invariants: a plan written onto a `created` project skipped every plan check until the status flipped. Validate whatever records exist on disk; use status only to decide what is *required*, never what is *checked*.
- 2026-07-06: A must-acknowledge rule keys on the finding that triggers it, not on the record's top-level status. The slice 4 advisory first warned only on `approval_status: block`, so an unwaived blocking-severity finding under a `revise` status passed silently. When severity carries the obligation, enforce on severity; the status is the reviewer's summary, not the contract.
- 2026-07-06: When per-slice reviews run concurrently with the next slice's edits, review fixes and new work collide on shared files and force edit-backouts to keep commits honest. Either hold the next slice's edits to files the pending review cannot touch, or wait for the review before opening shared files.
- 2026-07-06: A safety config parameter is an injection surface: dispatch honored an injected `config` for the kill switch, so a caller could pass `DISABLE_PAID_CONNECTORS: False` and re-enable paid calls the environment had disabled. Hard stops must be read from the real environment at the enforcement point; injected config may restrict further, never re-enable.
- 2026-07-06: A record a gate trusts must be bound to the place it sits, not just schema-valid: dispatch initially validated an approval record's shape and status but not that it targeted the project directory it lived in, so a hand-written record for another project dispatched fine. Every gate that loads a record by path must re-check the record↔location binding with the same shared function the writer used.
- 2026-07-06: Single-use consumption needs two phases and a lock: flipping approved→executed after the work means a crash or concurrent dispatch replays the work. Flip to an in-flight state (executing) under an O_EXCL lock before the first side effect; a leftover in-flight record refuses re-runs and surfaces as a warning instead of silently allowing a replay.
- 2026-07-06: A flat-directory reconciliation walk silently ignores nested files: `iterdir()` plus name-matching let `generation/assets/subdir/х` exist invisible to the ledger. When a validator promises "every file has a row", it must either walk recursively or fail closed on subdirectories — matching what the schema can represent.
- 2026-07-06: A gate keyed on an id trusts whatever currently wears the id: quality reviews scoped by asset_id alone kept covering artifacts whose bytes were replaced, and an old pass outlived a newer fail. Bind judgment records to the content hash they judged, and when several current-content judgments exist, let the latest decide.
- 2026-07-06: A conditional obligation invites the condition to be lied about: media provenance refs were required only when generation_status left planned_not_generated, so declaring planned_not_generated detached a ledgered project from every binding. When an obligation keys on a self-declared status, validate the status against the durable evidence (ledger rows) that makes it true or false.
- 2026-07-07: A plan that adds one value to "the platform enum" is really editing a canonical constant with pinned copies. The YouTube connector plan named 5 schemas, but the drift check pins every platform/content-type enum copy (~20 schemas), the code constants, and the platform-fit map to one canonical list — and the approval-surface gate started enforcing `youtube_*` distribution surfaces the moment `youtube` became a research platform (209 scaffold errors from one example promotion). When a plan touches a closed vocabulary, derive the edit list from the drift check's pinned-schema tables, not from the plan's file list, and expect dormant gates keyed on set membership to activate.
- 2026-07-07: A counter incremented per attempt reports success the operation never had. `enriched_count` counted enrichment attempts, so the fetch result claimed 12 enriched candidates while reddit.com answered every read with HTTP 403 and the helper silently returned items unchanged — mocks never exercised the failure, and the count validated fine. Counters that summarize an external side effect must count the observed effect (the key actually attached), and a helper that swallows errors must leave a countable absence its caller reports.
- 2026-07-07: Piecewise-correct validators compose into a false green when nothing runs them together. `validate project` trusted the promotion gate, which never re-resolved the queue entry's own evidence refs, so a project whose upstream evidence dangled validated in isolation; four scoped commands each passed their layer while the chain was broken. When validation is layered, the release claim must be one composed command (`validate all`) and the false green must be pinned by a test that passes the scoped command and fails the composed one.
- 2026-07-07: When a dormant gate activates on set membership, sweep the prose that taught the old behavior. Adding `youtube` to the research platforms activated the approval-surface check for `youtube_*` targets, but README and ARCHITECTURE kept telling operators YouTube Shorts was exempt — the code moved and the sentences describing it did not. A change that flips a documented behavior needs a grep for the behavior's phrasing ("exempt", "remains valid"), not only for the constant's name.
- 2026-07-07: Every confirmed defect in the 28-skill review lived inside duplicated prose — a template list restated outside its library, interview vocabulary leaked into a record contract, a path restated away from its schema pin — and zero lived in single-sourced material. Duplication is not a style cost; it is where drift physically happens, because a one-place edit cannot reach the second copy. When prose must be duplicated by design (runtime skill copies travel without docs/), enforce byte-identity with a drift test so a protocol change is a verified sweep, and otherwise replace restatements with pointers to the owning section.
- 2026-07-07: Skill frontmatter is code with a parser, not prose. Half the fleet's descriptions were invalid strict YAML (unquoted `: ` mid-scalar) and the lenient loader masked it for months — one toolchain change from silently losing every affected skill's invocation surface. In a stdlib-only repo, lint the observed failure shapes (quote balance, unsafe leading indicators, `: ` and ` #` in plain scalars) since a real parser is unavailable, and validate prose claims the same way as code: paths exist, vocabularies match schema enums, CLI snippets name real subcommands.
- 2026-07-09: A stage-gated validator can leave record invariants unenforced at earlier stages. Readiness milestone approval metadata, channel publish-state consistency, and `prompt_ready` media permissions must be checked before the lifecycle early-return; stage branches should add only the requirements that become mandatory at that stage.
- 2026-07-09: Renaming prose while retaining old nested field names does not resolve a domain-language conflict. When `CONTEXT.md` reserves a term such as Gate, sweep schema keys, constructor helpers, variables, headings, tests, and skills; retain legacy vocabulary only where compatibility actually requires it, such as an established filename.
- 2026-07-09: Schema-valid prompt records can still violate semantic cardinality. Adira's single object asset passed structural validation while its prompt explicitly requested eight props in one image. When a prose-authored artifact needs exactly one subject, enforce the fan-out rule at planning, repeat the invariant in the canonical template and completion criteria, and pin those owning phrases with a drift test.
- 2026-07-09: Atomic prompts do not prevent an oversized or incoherent reference library. Candidate props, product/brand objects, and production spaces need an evidence-resolving selection record, brand-effectiveness analysis, and explicit user decisions before asset promotion; lifecycle status should decide when approval is required, while record invariants remain validated at every stage.

## 2026-07-09 — Separate Visual Inspiration From Production Brand Systems

A generated brand image can communicate mood and layout direction while still
being unusable as a production contract. Creator setup must establish exact
palette, typography, imagery, and template tokens in structured data, render
them through one reusable editable template, and collect approval at that
specific artifact boundary.

Reusable board sections must have one stable semantic meaning. Production
spaces link to location assets; signature props link to object assets; content
pillars remain typographic unless an image directly depicts the pillar.

When a visual document depends on reusable assets, sequence it after asset
planning and link by typed identity rather than copying file paths. This makes
wrong-category images fail validation, supports prompt-ready placeholders, and
makes the projection stale when its source asset records change.

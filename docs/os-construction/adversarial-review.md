## Executive Summary

InfluencerOS is a faithful, unusually well-documented adaptation of agentic-os at the architecture level — the conductor-plus-skills orchestration, creator-vs-OS memory scope split, Creator Workspace boundary, project-folder consolidation, and progressive-disclosure principle are all preserved and, in most cases, explicitly recorded as accepted divergences with ADRs and alignment-doc entries. The repo is genuinely in a pre-implementation Phase 0 (parity hardening), so most gaps are known, tracked, and gated behind open decisions rather than silent drift. That said, three issues rise above the rest. First, the self-improvement loop is non-functional: only the destination files (context/learnings.md, process-learnings.md) were copied, never the wrap-up/learning-write mechanism that writes them, so the OS cannot improve its own skills from feedback — this is the owner's #1 priority and it is dark in v1. Second, determinism erodes precisely where the product's own bar is highest: the Level 2 brief's acceptance-criteria/constraints/dependencies contract was silently dropped from the Project schema (the single HIGH finding), the Output Package under-enforces the prose Product Invariant, provenance IDs are pattern-checked but never resolved, and named pipeline boundaries (Format Shortlist) have no output contract. Third, the file-granular orchestrator→conductor→subskill call graph the owner wants mapped does not exist — the conductors declare their call graph only in prose, the flagship content conductor routes to six skills that do not exist, and the doc labeled "Creation Flow Call Map" is actually a record data-flow map. Underpinning all of this: all six PRD-promised drift checks are unimplemented, so the hand-maintained-docs model InfluencerOS chose over agentic-os's generate-from-catalog model has no automated guard and can silently go stale today.

## Cross-Cutting Themes

- Destination-without-mechanism: repeatedly InfluencerOS copied the OUTPUT locations of an agentic-os subsystem (context/learnings.md, context/MEMORY.md, SKILL.local.md convention, workspace directories) without copying the SKILL/script that writes or maintains them — the self-learning loop, Tier-0 memory writes, and skill self-correction are all dead files for this reason.
- Prose vs. machine-actionable contract: the highest-value gaps are where a guarantee is asserted in prose (AGENTS.md Product Invariant, conductor call graphs, 'same read order', TDD/CDD/DDD, phase exit criteria) but never encoded in a schema, validator, dependency declaration, or test — so nothing can fail when the prose stops being true.
- Determinism enforced downstream, absent upstream: the production chain (Selected Idea -> Output Package) is genuinely stronger than agentic-os with status-gated records, const gates, and ID-equality checks, but the upstream/mid-pipeline half (Research -> Format Shortlist -> Idea Set -> Selection) and the Project brief semantics are unvalidated, unresolved, or entirely missing a contract.
- Provenance is pattern-checked, not resolved: across projects and mid-pipeline boundaries, IDs are validated as regex strings and cross-checked for equality between co-located records, but no code resolves them to real source files/records, so the advertised trace-back guarantee accepts dangling references.
- No automated guard for a hand-maintained-docs architecture: InfluencerOS deliberately diverged from agentic-os's generate-from-catalog model to hand-written parallel docs (registry, matrix, three adapter read-orders) but has built none of the six PRD-promised drift checks, so registry/matrix/adapter/schema drift is invisible and the promised checks are specified too vaguely to implement test-first.
- Documented-and-deferred vs. silently-omitted: most gaps are honestly tracked as open Phase 0C decisions (skill layout, workspace propagation, memory recall), which is the governing rule working as intended — but a few are genuinely silent (the entire .claude/agents/ subagent subsystem was never inspected; the dropped brief acceptance-criteria contract), and self-contradicting docs (skill layout accepted-vs-open, client sync rejected-vs-needs-decision, adapters claim a shared read order they lack) blur which is which.

### [1] Level 2 brief's acceptance-criteria / constraints / dependencies contract is silently dropped from the Project schema

**🔴 HIGH** · dimension: `projects-output` · type: `divergence` · verdict: CONFIRMED (high confidence)

- **Agentic OS reality:** agentic-os defines a Level 2 project as a `brief.md` with an explicit five-part contract. AGENTS.md:297 states: "Level 2 brief requirements: goal, deliverables, acceptance criteria, constraints, and dependencies. Keep it to one page." README.md:431 shows the project folder centered on `brief.md`, and AGENTS.md:293-295 makes the brief the defining artifact of a Level 2/3 project. Acceptance criteria are the deterministic 'done' condition for the whole project.
- **InfluencerOS reality:** InfluencerOS replaces `brief.md` wholesale with schemas/project.schema.json. That schema's required fields are project_id, creator_profile_id, project_slug, created_on, status, content_unit_type, source_refs, project_paths, target_formats, learning_goal (schemas/project.schema.json:7). It captures GOAL (as `learning_goal`, line 41) and partial DELIVERABLES (as `target_formats`, line 39), but there is NO field for acceptance criteria, constraints, or dependencies. examples/project.example.json confirms this — the only goal-like content is `learning_goal` plus a freeform `notes` string. The agentic-os copy-plan (docs/os-construction/agentic-os-copy-plan.md:66) documents the consolidation divergence only as "schema-backed Output Packages and provenance, not date-named markdown deliverables" — it never mentions that the brief's acceptance-criteria/constraints/dependencies semantics are being dropped. maps/agentic-os-vs-influencer-os.md:73 similarly frames it only as markdown-vs-schema.
- **Why it matters:** Acceptance criteria are the load-bearing part of the agentic-os brief: they are the project's explicit, verifiable success condition. CLAUDE.md's non-negotiable is "Keep every workflow deterministic at the boundary: explicit inputs, schema-backed outputs, acceptance criteria, and validation." A Project with no acceptance-criteria field means the completion bar for a whole content unit lives nowhere structured — an agent cannot check the project against its own definition of done, and validate_project (influencer_os/projects.py) can only confirm files exist and IDs match, never that the project met its stated bar. This is a semantic regression from agentic-os that was not surfaced or approved as a divergence, so it fails the repo's own divergence-test (docs/os-construction/divergence-test.md:11-19).
- **Recommendation:** Add a required acceptance-criteria construct to project.schema.json (e.g. an `acceptance_criteria` array of checkable statements, plus optional `constraints` and `dependencies`) so the Project record carries the same completion contract as an agentic-os Level 2 brief. If InfluencerOS intends to fold acceptance criteria into per-format production plans instead, record that as an explicit documented+approved divergence in agentic-os-alignment.md or an ADR — right now it is neither present nor documented.
- **Verifier note:** All factual claims verified independently in both repos. (1) agentic-os read correctly: AGENTS.md:297 requires the Level 2 brief to carry goal, deliverables, acceptance criteria, constraints, and dependencies; README.md:431 centers the project folder on brief.md. (2) InfluencerOS read correctly: schemas/project.schema.json required fields (line 7) are project_id, creator_profile_id, project_slug, created_on, status, content_unit_type, source_refs, project_paths, target_formats, learning_goal — with only optional platform_targets/notes. There is genuinely NO field for acceptance criteria, constraints, or dependencies; learning_goal (line 41) captures GOAL and target_formats (line 39) captures partial DELIVERABLES, matching the finder. examples/project.example.json confirms. I also checked every plan schema (base-video-generation-plan, micro-journey, carousel, single-image-post, story-sequence): none contain constraint/dependency/acceptance fields, so the semantics are not relocated to the plan stage. A repo-wide grep shows "acceptance criteria" exists only as an OS-construction concept (PRD, short-term-plan, CLAUDE.md boundary rule) — never as a project-level content contract field. (3) The divergence is silent: agentic-os-copy-plan.md:66 frames the Output-consolidation divergence only as "schema-backed Output Packages and provenance, not date-named markdown deliverables"; maps/agentic-os-vs-influencer-os.md Outputs row frames it only as markdown-first-vs-schema-backed; ADR 0012 defines the project folder layout (idea/plan/output-package/published/analytics) but never addresses the brief's acceptance/constraints/dependencies contract; roadmap "approved standing overrides" (lines 17-25) do not cover it; no ADR (0001-0014) approves dropping it. So it fails the repo's own divergence-test.md:11-19 (no alignment-doc entry, no approving ADR). (4) validate_project (influencer_os/projects.py:65-97) only checks manifest schema-validity, root/slug match, and file existence — it can never confirm a project met a stated done-bar, exactly as claimed. Severity HIGH is justified: InfluencerOS Projects are unambiguously the planned multi-deliverable tier (idea through analytics, multi-platform packages, with a stated learning_goal), which is precisely the Level 2/3 case agentic-os requires a full brief for; dropping acceptance criteria removes the verifiable success condition that CLAUDE.md names as a non-negotiable ("explicit inputs, schema-backed outputs, acceptance criteria, and validation"). The only minor nuance — that agentic-os requires no brief for Level 1 single-task work — does not lower impact, because these Projects are not Level 1.

### [2] The self-improvement MECHANISM (a wrap-up/learning-write skill) is entirely absent from v1 — only its destination files were copied

**🟠 MEDIUM** · dimension: `self-learning` · type: `gap` · verdict: ADJUSTED (high confidence)

- **Agentic OS reality:** agentic-os ships the loop as a MODEL-INVOKED SKILL, not as automation. `.claude/skills/meta-wrap-up/SKILL.md` is a full mechanism: Step 3a appends per-skill entries to `context/learnings.md`, Step 3b edits SKILL.md directly, Step 3c finalizes daily memory, 3d evolves SOUL.md, 3e updates USER.md, 3f reconciles the Skill Registry, 3g promotes to MEMORY.md, Step 4 commits. It triggers off its own `description` (session-end phrases / `/wrap-up`) — it is NOT listed in `.claude/settings.json` hooks (grep for 'wrap-up' there is empty). `.claude/skills/meta-memory-write/SKILL.md` is the companion MEMORY.md-write mechanism. `context/learnings.md` is populated with real dated per-skill entries (e.g. mkt-brand-voice 2026-06-29) written BY that skill.
- **InfluencerOS reality:** No wrap-up, distill, or memory-write skill exists anywhere on disk. `find skills -type d` returns only 1 conductor (`influencer-os`) + `create-influencer` + 7 `create-*` setup subskills; a `find . -name 'SKILL.local.md'` returns nothing. `grep -niE 'wrap-up|learnings|self-update|feedback' skills/influencer-os/SKILL.md` and `skills/create-influencer/SKILL.md` return NOTHING — no skill has a step that writes learnings. The destination files were copied but not their writer: root `context/learnings.md` and `docs/os-construction/process-learnings.md` are empty scaffolds with placeholder text ('Pending repeated feedback.'). So InfluencerOS copied the OUTPUT LOCATIONS of the loop without copying the SKILL that produces the output.
- **Why it matters:** This is the highest-priority dimension and the loop is non-functional in v1. Without a wrap-up/learning-write skill, `context/learnings.md`, `process-learnings.md`, and the SKILL.local.md override pipeline are dead files — nothing ever appends to them, so InfluencerOS cannot improve its own skills from feedback. Every ADR that assumes the promotion pipeline (0014's 'promote a local rule after repeated feedback', the process-learnings 'Promotion Rule') is unenforceable because step 1 (capture feedback) has no executing agent.
- **Recommendation:** Adopt the agentic-os pattern now: copy `meta-wrap-up` and `meta-memory-write` into `skills/` as system-level skills, adapted to InfluencerOS paths (root `context/learnings.md`, `docs/os-construction/process-learnings.md`, Skill Registry reconciliation instead of AGENTS.md). This is pure MECHANISM and requires no hooks or cron — it triggers off the skill description exactly as in the template. Do NOT wait for Phase 2; the system-level learning skill is Phase-independent.
- **Verifier note:** The finding's facts are accurate on both sides. agentic-os `.claude/skills/meta-wrap-up/SKILL.md` is verified as a full model-invoked skill (Steps 3a-3h write context/learnings.md, edit SKILL.md, finalize daily memory, evolve SOUL.md, reconcile the registry, promote to MEMORY.md; Step 4 commits) triggered by its own description, not a settings hook. In InfluencerOS, `find skills -type d` returns only the influencer-os conductor plus create-influencer and 7 create-* setup subskills; no SKILL.local.md exists; and `grep -niE 'wrap-up|learnings|feedback|distill|memory-write'` over skills/influencer-os/SKILL.md and skills/create-influencer/SKILL.md returns nothing. The destination files are placeholder scaffolds: /Users/ashokaji/code/fullstock/InfluencerOS/context/learnings.md and docs/os-construction/process-learnings.md both say "Pending repeated feedback." ADR 0014's promotion pipeline and the process-learnings "Promotion Rule" both begin with "capture feedback," which has no executing agent. So the writer mechanism is genuinely absent while only its output locations were copied — the finding is real.

However, HIGH overstates it because this is NOT a silent divergence in a shipping product: (1) docs/os-construction/agentic-os-alignment.md marks the self-improvement row "Contracted" (explicitly planned/not-yet-built), not "Adopted"; (2) roadmap.md shows the repo is in Phase 0 (parity hardening), and agentic-os-copy-plan.md row 63 says "do not implement new workflow skills beyond parity hardening" until a Phase 0C skill-layout decision is made; (3) the empty scaffolds are placeholders by design, consistent with that status. The self-learning approach itself is documented and "accepted" (copy-plan row 73; ADR 0014 Accepted), so the architecture is approved — only the executing skill is unbuilt, which is expected at Phase 0. The residual real issue is a planning-completeness gap, not a functional defect: no roadmap phase clearly owns building the wrap-up/learning-write skill — Phase 2 "Learning OS" is scoped to creator PERFORMANCE learning (Published Post Records, Analytics Snapshots, Performance Summaries, Creator Memory), not the SKILL self-improvement loop, and Phase 0C concerns skill layout, not the writer. That under-specification is a legitimate MEDIUM finding; HIGH is inflated relative to a v1 that never claimed the loop was operational.

## Self-Learning Gap Analysis

The self-improvement loop is the owner's #1 priority and it is non-functional in v1 for a single, clean reason: agentic-os ships the loop as a MODEL-INVOKED SKILL (meta-wrap-up + meta-memory-write), NOT as automation — those skills write context/learnings.md, edit SKILL.md directly on feedback, finalize daily memory, and promote durable facts to MEMORY.md, and they trigger off their own skill description, never a hook. The Stop-hook and the nightly cron are explicitly redundancy layered on top of that manual agent layer, which the hook's own header states. InfluencerOS copied the destinations (empty context/learnings.md and process-learnings.md scaffolds reading "Pending repeated feedback", the SKILL.local.md override convention, the promotion rule in ADR 0014) but copied neither writer skill and added no Returning-Mode/wrap-up contract to CLAUDE.md/AGENTS.md — so learnings.md never fills, SKILL.local.md is never written, and ADR 0014's "promote a local rule after repeated feedback" pipeline can never fire because step one (capture feedback) has no executing agent. Critically, the loop IS recoverable local-first WITHOUT hooks: it is pure mechanism, needs no cron, PGLite, or Command Centre, and triggers off the skill description exactly as in the template. The recommended v1 shape: (1) copy/adapt meta-wrap-up and meta-memory-write into skills/ as system-level skills wired to InfluencerOS paths (root context/learnings.md, process-learnings.md, Skill Registry reconciliation), (2) add a Returning-Mode startup + wrap-up step to CLAUDE.md/AGENTS.md that reads MEMORY.md and appends per-skill learnings and daily session blocks manually, (3) add the ## Rules + ## Self-Update section pattern plus one worked SKILL.local.md example to the two behavior-changing skills so the override load-order is exercisable, and (4) enforce the 2.5 KB MEMORY.md cap with an explicit pre-write wc -c gate. Do NOT fold this into Phase 2: Phase 2 "Learning OS" is correctly scoped to creator PERFORMANCE learning (distill-creator-learning), which genuinely depends on unbuilt Output Package/Analytics schemas — but the SYSTEM skill-learning loop has no such dependency and was incorrectly swept into that deferral.
